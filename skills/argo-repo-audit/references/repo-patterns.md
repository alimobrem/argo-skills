# Argo CD Repository Patterns

## Classification Heuristics

Use these heuristics with the discovery output to classify the repository pattern.
A repo may exhibit multiple patterns; report the primary one and note secondary patterns.

---

## App of Apps

### Heuristic

- Application resources exist whose `spec.source.path` points to a directory **within the same repo** that contains other Application YAML files.
- Typically a root Application in `apps/` or `bootstrap/` that points to a directory of child Applications.
- The root Application acts as the single entry point for Argo CD to manage.

### Structure

```
repo/
  projects/
    team-a.yaml          # AppProject
  apps/
    root-app.yaml         # Application -> spec.source.path: apps/children
    children/
      app-a.yaml          # Application -> spec.source.path: manifests/app-a
      app-b.yaml          # Application -> spec.source.path: manifests/app-b
  manifests/
    app-a/
      deployment.yaml
      service.yaml
    app-b/
      deployment.yaml
      service.yaml
```

### Pros

- Single entry point — sync one Application to bootstrap everything
- Git-native dependency management via sync waves
- Full visibility in Argo CD UI (each child is its own Application)

### Cons

- Boilerplate — each new app requires a new Application YAML
- Deletion cascading requires careful finalizer management
- Can become deeply nested (app of apps of apps)

## ApplicationSet

### Heuristic

- ApplicationSet resources with `spec.generators` (list, git, cluster, matrix, merge, pull request, SCM provider).
- No manually-created Application resources (they are generated by the ApplicationSet controller).
- Often paired with a `clusters/` or `config/` directory holding generator input data.

### Structure

```
repo/
  applicationsets/
    platform-apps.yaml    # ApplicationSet with git directory generator
    team-apps.yaml        # ApplicationSet with list generator
  clusters/
    staging/
      config.json         # Generator parameters
    production/
      config.json
  apps/
    podinfo/
      kustomization.yaml
    nginx/
      kustomization.yaml
```

### Pros

- DRY — one ApplicationSet replaces N Application manifests
- Scales to hundreds of clusters/environments
- Progressive syncs (rollingSync) for safe fleet rollouts
- Auto-discovery via git/SCM generators

### Cons

- Debugging generated Applications harder than static ones
- Template errors affect all generated Applications
- Matrix/merge generators can produce unexpected combinations
- Requires ApplicationSet controller (included in Argo CD 2.3+)

## Monorepo

### Heuristic

- All Applications reference the **same `repoURL`** (the repo itself).
- Different environments distinguished by `spec.source.path` pointing to Kustomize overlays.
- Overlay directories named by environment (e.g., `overlays/staging`, `overlays/production`).
- May or may not use app-of-apps or ApplicationSet on top.

### Structure

```
repo/
  apps/
    podinfo/
      base/
        deployment.yaml
        service.yaml
        kustomization.yaml
      overlays/
        staging/
          kustomization.yaml    # patches for staging
        production/
          kustomization.yaml    # patches for production
    nginx/
      base/
        deployment.yaml
        kustomization.yaml
      overlays/
        staging/
          kustomization.yaml
        production/
          kustomization.yaml
  argocd/
    staging-podinfo.yaml    # Application: source.path = apps/podinfo/overlays/staging
    prod-podinfo.yaml       # Application: source.path = apps/podinfo/overlays/production
```

### Pros

- Single source of truth — all manifests in one place
- Atomic commits across multiple apps/environments
- Easy to review cross-cutting changes

### Cons

- Large repos slow down Argo CD reconciliation
- Any commit triggers all Applications to reconcile (mitigated by path filters)
- Permissions harder to scope (one repo = one set of access controls)

## Multi-Repo

### Heuristic

- Applications reference **different `repoURL`** values.
- Typically a "control plane" repo with Application/AppProject definitions pointing to separate "payload" repos.
- Each team/service owns its own repo.

### Structure

```
control-repo/                     # This repo
  projects/
    team-a.yaml                   # AppProject: sourceRepos restricted to team-a-repo
  apps/
    team-a-app.yaml               # Application: repoURL = https://github.com/org/team-a-repo
    team-b-app.yaml               # Application: repoURL = https://github.com/org/team-b-repo

team-a-repo/                      # Separate repo
  manifests/
    deployment.yaml
    service.yaml

team-b-repo/                      # Separate repo
  charts/
    my-chart/
      Chart.yaml
      values.yaml
```

### Pros

- Team autonomy — each team controls their own repo
- Fine-grained access control via AppProject sourceRepos
- Independent release cycles

### Cons

- Cross-cutting changes require coordinated PRs
- Harder to audit full system state
- Repository credential management scales linearly with repos

## Environment Branch

### Heuristic

- Applications reference the **same `repoURL`** but **different `targetRevision`** values.
- Branch names correspond to environments (e.g., `staging`, `production`, `main`).
- Manifests at the same path in different branches represent different environment configs.

### Structure

```
repo (main branch = staging):
  apps/
    podinfo/
      deployment.yaml       # staging config
      service.yaml

repo (production branch):
  apps/
    podinfo/
      deployment.yaml       # production config (different replicas, resources, etc.)
      service.yaml

control-repo:
  apps/
    staging-podinfo.yaml    # Application: targetRevision = main
    prod-podinfo.yaml       # Application: targetRevision = production
```

### Pros

- Clear promotion path (merge staging -> production)
- Environment isolation at the Git level
- Familiar branch-based workflow

### Cons

- Branch drift — environments can diverge silently
- Cherry-pick/merge conflicts between long-lived branches
- Harder to diff environments (requires cross-branch comparison)
- Generally considered an anti-pattern — Kustomize overlays preferred

## Hybrid Patterns

Real-world repos often combine patterns:

| Combination | Example |
|-------------|---------|
| App of Apps + Monorepo | Root app bootstraps child apps, all in same repo with Kustomize overlays |
| ApplicationSet + Monorepo | ApplicationSet git generator discovers apps in a monorepo |
| ApplicationSet + Multi-Repo | ApplicationSet SCM provider generator discovers repos across an org |
| App of Apps + Multi-Repo | Root app in control repo, child apps point to separate payload repos |

When classifying, report the **primary orchestration pattern** (how Applications are created) and the **primary source pattern** (where manifests live) separately if they differ.
